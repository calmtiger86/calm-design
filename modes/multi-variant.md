# Mode D: Multi-Variant — 3안 동시 생성

calm-design의 4가지 모드 중 마지막. 한 사용자 요청에 대해 **3개의 다른 디자인 베리언트**를 동시 생성하여 사용자가 선택·발전할 수 있게 함. 디자인 에이전시의 "mood board" 워크플로우 모방.

> Mode A(Generate)는 단일안, Mode D는 3안. 핵심 차이: 의도적으로 **다른 카테고리의 패턴**을 채택하여 다양성 확보.

## 트리거 키워드

- "3가지 안", "여러 옵션", "다양하게"
- "베리언트", "변형", "옵션 보여줘"
- "다른 톤도", "옵션 비교"
- "moodboard", "mood board"

> 다른 모드 키워드와 충돌 시 **Mode D 우선** (가장 높은 우선순위)

## 10단계 워크플로우

### Step 0. CONTEXT.md 로드 (선택)

`.calm-design/CONTEXT.md` 존재 시 로드:
- `suggested_dials` → V3 (도메인 매칭) 베리언트의 기본 다이얼로 사용
- `design_mappings.color` → 3개 베리언트 모두의 감정 톤 힌트로 활용
- `anti_patterns.project_specific` → 모든 베리언트에 공통 적용
- `memorable_point` → V1의 Hero 카피에 반영
- CONTEXT.md 활용 시 "📋 CONTEXT.md 반영됨" 표시

### Step 1. 사용자 입력 분석 + 공통 의도 추출

Mode A의 5축 분류기를 활용하되, **공통 의도**만 추출 (페이지 타입·언어·모드). 다이얼은 베리언트별로 다를 것이므로 무시.

```yaml
common_intent:
  page_type: landing
  language: ko
  brand_constraints:
    keep: ["로고", "Toss Blue 액센트"]   # 사용자가 명시한 보존 조건
```

### Step 2. 3개 베리언트 매트릭스 결정

`references/creative-arsenal.md` Mode D 매트릭스 + `references/color-system.md` 컬러 매트릭스 + `references/motion-system.md` 모션 매트릭스를 **동시 sparse load**.

#### 표준 3-베리언트 매트릭스

| | **V1: 미니멀 차분** | **V2: 트렌디 다이내믹** | **V3: 도메인 매칭** |
|---|---|---|---|
| **VARIANCE** | 4 | 8 | (Mode A 추론값) |
| **MOTION** | 4 | 7 | (Mode A 추론값) |
| **DENSITY** | 4 | 5 | (Mode A 추론값) |
| **컬러 베이스** | Stone (따뜻 중립) | Zinc (차가운 중립) | (사용자 의도 기반) |
| **액센트** | Calm Emerald | Warm Amber 또는 Deep Rose | (사용자 의도 또는 Toss Blue) |
| **레이아웃** | Asymmetric Whitespace | Bento Grid + Inline Image Typography | (페이지 타입 표준) |
| **모션 톤** | Hover만 + Fade-up | Magnetic Button + Counter Up | Stagger Cascade + Spring |
| **타이포** | Pretendard (한국어 정통) | Cabinet Grotesk + Pretendard (한·영 페어링) | Pretendard 표준 |
| **시그니처** | (없음 — 절제) | Inline Image Typography | (도메인 의존) |

이 매트릭스는 절대 변경 금지. 3개 베리언트가 시각적으로 명백히 달라야 함.

### Step 3. 보존 조건 → 3개 베리언트 모두 강제 적용

사용자가 "Toss Blue는 유지", "로고는 그대로" 같은 보존 조건을 명시하면 모든 베리언트에 강제 적용:

- V1·V2·V3 모두 Toss Blue 액센트 유지
- 단, 액센트 외 영역(베이스 컬러·레이아웃·모션)은 베리언트별로 다르게

→ 보존 조건과 매트릭스가 충돌하면 **보존 조건 우선** (V2의 Warm Amber 액센트 → Toss Blue로 교체).

### Step 4. 3개 DESIGN.md 병렬 생성

각 베리언트마다 `references/design-md-spec.md` 9-섹션 표준 적용:

```
.calm-design/
├── variants/
│   ├── v1-minimal/
│   │   └── DESIGN.md
│   ├── v2-trendy/
│   │   └── DESIGN.md
│   └── v3-domain/
│       └── DESIGN.md
└── comparison.md   ← Step 8에서 생성
```

각 DESIGN.md 헤더에 베리언트 메타 명시:

```markdown
> Generated by calm-design · LANGUAGE: ko · VARIANCE: 4 · MOTION: 4 · DENSITY: 4 · Mode: D · Variant: V1 (미니멀 차분)
```

### Step 5. 출력 변환 (각 베리언트별)

사용자가 명시한 출력 엔진 적용. 기본은 HTML. 3개 다 만들면 시간 부담 → Phase 2.0은 HTML만, JSON Spec은 옵션.

3개 베리언트의 HTML 산출물:

```
.calm-design/variants/
├── v1-minimal/
│   ├── DESIGN.md
│   └── index.html
├── v2-trendy/
│   ├── DESIGN.md
│   └── index.html
└── v3-domain/
    ├── DESIGN.md
    └── index.html
```

### Step 6. 셀프-크리틱 (각 베리언트 독립)

3개 베리언트 모두 `references/self-critique-loop.md` 5단계 통과 의무. 한 베리언트라도 ❌ 잔존 시 그 베리언트만 재생성.

```yaml
v1-minimal:    Pre-Flight 30/30 ✅
v2-trendy:     Pre-Flight 28/30 ⚠️ (재생성 1회 → 30/30)
v3-domain:     Pre-Flight 30/30 ✅
```

### Step 7. SpecStream 점진 표시 (json-render 결합 #3)

JSON Spec 출력 시 SpecStream 컴파일러 활용. 3개 베리언트가 점진적으로 사용자에게 표시:

```
[T+0s]    V1 골격 등장
[T+1s]    V2 골격 등장
[T+2s]    V3 골격 등장
[T+5s]    각 베리언트 색상·타이포 채워짐
[T+10s]   각 베리언트 컴포넌트 디테일 채워짐
[T+15s]   3개 베리언트 모두 완성 + 비교 리포트
```

> 사용자는 "기다리는 화면" 대신 점진적으로 채워지는 3개 캔버스 관찰. calm-design "Calm" 정체성과 일치.

### Step 8. 비교 리포트 자동 생성 (`comparison.md`)

3개 베리언트의 차이를 한눈에:

```markdown
# 🎨 calm-design Mode D — 3안 비교 (2026-04-26)

| 차원 | V1 미니멀 차분 | V2 트렌디 다이내믹 | V3 도메인 매칭 |
|---|---|---|---|
| 다이얼 | 4·4·4 | 8·7·5 | 7·6·4 |
| 베이스 | Stone (따뜻) | Zinc (차가움) | Stone-50 + Toss Blue |
| 액센트 | #10B981 Emerald | #F59E0B Amber | #3182F6 Toss Blue |
| 레이아웃 | Asymmetric WS | Bento + Inline Img Type | Split Hero + Bento |
| 시그니처 | (없음) | Inline Image Typography | Toss Bottom Sheet 권장 |
| 모션 강도 | 미세 (Hover만) | 영화적 (Magnetic+Counter) | 표준 (Stagger Cascade) |
| 한국어 톤 | 정통 Pretendard | 한·영 페어링 (Cabinet+Pretendard) | Pretendard |
| Pre-Flight | 30/30 ✅ | 30/30 ✅ | 30/30 ✅ |
| 추천 시나리오 | B2B SaaS, 금융, 신뢰 우선 | 마케팅 랜딩, 크리에이터 | 사용자 명시 도메인 |

## 어떤 베리언트를 선택하시겠어요?

각 베리언트의 [DESIGN.md](variants/v1-minimal/DESIGN.md) + [HTML](variants/v1-minimal/index.html)
을 검토 후, 다음 중 선택:

1. **V1 채택** — 그대로 진행
2. **V2 채택** — 그대로 진행
3. **V3 채택** — 그대로 진행
4. **혼합** — 예: "V1 컬러 + V2 레이아웃" → 새 베리언트 생성
5. **모두 거부** — 새 매트릭스로 재시도
```

### Step 9. 산출물 6가지 사용자 전달

```markdown
## ✅ calm-design Mode D 완료

[ V1 DESIGN.md ]              .calm-design/variants/v1-minimal/DESIGN.md
[ V2 DESIGN.md ]              .calm-design/variants/v2-trendy/DESIGN.md
[ V3 DESIGN.md ]              .calm-design/variants/v3-domain/DESIGN.md
[ 각 베리언트 출력 코드 ]    HTML 또는 JSON Spec (선택한 출력 엔진)
[ comparison.md ]             3안 차원별 비교
[ 셀프-크리틱 통합 리포트 ]  3개 베리언트 모두 30/30 통과 확인
```

## 보존 조건 처리 매트릭스

| 보존 조건 | V1 | V2 | V3 |
|---|:-:|:-:|:-:|
| Toss Blue 액센트 유지 | ✅ V1의 Emerald → Toss Blue 교체 | ✅ V2의 Warm Amber → Toss Blue 교체 | ✅ |

| Hero 카피 유지 | ✅ V1·V2·V3 모두 동일 카피 | ✅ | ✅ |
| 페이지 구조 유지 | ✅ 7섹션 동일 순서 | ✅ | ✅ |

## Mode D 사용 예시

```
사용자: "한국 SaaS 랜딩페이지 3가지 안 보여줘. Toss Blue는 유지."

→ Step 1: 페이지=landing, 언어=ko, 보존={Toss Blue}
→ Step 2: 매트릭스 적용
   V1 (미니멀): VARIANCE=4, MOTION=4, Stone 베이스, Toss Blue 액센트, Asymmetric WS
   V2 (트렌디): VARIANCE=8, MOTION=7, Zinc 베이스, Toss Blue 액센트(원래는 Amber지만 보존), Bento + Inline Img Type
   V3 (도메인): VARIANCE=7, MOTION=6, Stone-50 베이스, Toss Blue 액센트, Split Hero + Bento
→ Step 3: 보존 검증 통과
→ Step 4-7: 3개 DESIGN.md + HTML + 셀프크리틱
→ Step 8: comparison.md 생성
→ Step 9: 사용자에게 6가지 산출물 전달
```

## 한계·비용 (정직한 명시)

- **시간**: 단일 Mode A 대비 약 3배 + SpecStream 스트리밍 시 추가 지연
- **토큰 비용**: 셀프-크리틱 3회 = 단일 모드 3배
- **디스크**: variants/ 디렉토리에 3 × 산출물 생성
- 사용자가 `--variants=2`로 2안만 요청 시 V1+V2만 생성 가능

## 다른 모드와의 분기

Mode D 진행 중 다음 시그널 발견 시 자동 전환:
- "이거 그대로 더 다듬어줘" + V1 명시 → Mode B (Upgrade) 전환, V1만 처리
- "토스 스타일로 일괄" → Mode C (Match-Reference) 전환, 단일 베리언트로
